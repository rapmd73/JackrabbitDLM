## Section 1 - Non-Technical Description

This program manages access to shared resources in a distributed computing environment, ensuring that only one process can use a resource at a time through a locking mechanism. It also provides a way to temporarily store and retrieve small pieces of data associated with these resources, with automatic cleanup of expired locks and data. The system listens for requests over a network connection and responds to commands for locking resources, unlocking them, storing data, retrieving stored data, or removing data, with all operations having time limits to prevent permanent locks or data storage.

## Section 2 - Technical Analysis

The program implements a Distributed Lock Manager (DLM) that operates as a network service. It begins by setting up global constants including version information, directory paths, and size limits for data handling. The core data structure is the global Locker dictionary, which maintains all active locks and stored data items, with each entry containing an ID, expiration time, and potentially authentication details and stored data. The program initializes statistics counters to track various operations and sets memory constraints including MaxBlockSize (4096 bytes per read), MaxTTL (3543 seconds for authentication requirements), and MaxPayloadSize (10MB for payload limits).

The main() function initializes the network server by creating a non-blocking TCP socket that listens on port 37373 (configurable via command line arguments). It uses the poll() system call to efficiently manage multiple client connections. When a client connects, the server accepts the connection and registers it with the poller. For each connection, the server accumulates incoming data until a complete message (terminated by a newline) is received, at which point it processes the payload through the ProcessPayload function. The server tracks connection data in dataStore, sendBuffer, and queue dictionaries, and handles connection cleanup when complete.

The ProcessPayload function handles different request types by parsing the JSON input and performing the requested operation. For lock requests, it either creates a new lock entry if none exists, extends an existing lock if the requester is the owner, or rejects the request if another process holds the lock. Unlock requests verify ownership before releasing the lock. The get and put operations handle data retrieval and storage respectively, with put requiring a DataStore field in the payload. Erase requests remove stored data after ownership verification. Each operation updates statistics counters and returns appropriate JSON responses through the jsonStatus function. Authentication is verified using the CheckAuthentication function when required, particularly for operations with expiration times exceeding MaxTTL or when 'Name' is specified in the payload.

The main loop periodically checks for expired locks and data during each iteration, removing them from the Locker dictionary and tracking the memory usage of removed data in MaxDataHoldSize. It triggers garbage collection when this value exceeds MaxPayloadSize. Statistics about operations, connections, and system performance are logged hourly to a file through the WriteLog function. The program carefully manages memory by tracking data sizes from closed connections and triggering garbage collection when necessary, while also enforcing size limits on incoming payloads to prevent resource exhaustion. Each connection is handled independently with proper cleanup of resources when connections terminate normally or abnormally.
